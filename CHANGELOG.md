## [Unreleased]

### Changed
- **Administration Frontend Design System frozen** (`docs/ADMIN_DESIGN_SYSTEM.md`, new document): a dedicated reverse-engineering and design pass over the legacy admin (`alqla-main/admin-frontend`), analyzed as user journeys (first impression, CRUD, settings, administrative, mobile, accessibility, RTL) rather than screens, per explicit instruction to preserve experience, not screens. Ten concrete UX mistakes were found and documented with cause/impact/fix (a non-functional decorative search box; a notifications button that is a documented stub with no backing data; zero unsaved-changes protection anywhere, including on a form containing an unconfirmed site-wide maintenance-mode toggle; uniform confirmation-dialog weight regardless of action risk; among others). Specifies design tokens (resolving Old Admin's own self-contradiction — a Tailwind config forcing border-radius to zero everywhere while nearly every real interactive component overrides it to `rounded-xl`/`rounded-2xl` — by adopting New Admin's already-present ~6–10px radius scale uniformly instead), a full old→new component mapping table, and the one genuine architecture-adjacent gap found: New Admin's `WorkspaceDefinition` navigation model is flat, but the requested new information architecture needs one grouping level for Administration's nine child capabilities — closed by one additive, optional field, not a redesign. Three additional decisions frozen in the same pass: **Digital Experience** as the correct name for Administration's website-adjacent capability (Branding, Login Experience, Domains, SEO, Analytics, Tracking, Social Presence, PWA, Public Identity), resolving a naming collision with the unrelated top-level `Website` (CMS/Pages/Content) workspace; **Lucide Icons** as the sole, official icon library ERP-wide (evaluated against Tabler and Heroicons across ten criteria — decisive factors: already the icon library in both the legacy and new codebases, meaning zero loss of icon-shape continuity, plus tighter design consistency at this product's actual icon-vocabulary scale), with a frozen sizing/stroke/state system; and a fully specified **Login Experience** (school logo and background sourced from the Configuration Platform as real Media references, dark/light mode, optional background slider/video, bilingual Configuration-driven welcome copy and rotating messages, a maintenance-mode message distinct from the authenticated Diagnostics page, and a three-step Loading Experience → Workspace Bootstrap → Dashboard post-auth sequence). The Installation Wizard and First-Time School Setup Wizard are documented as two genuinely distinct experiences (technical/DevOps-facing infrastructure bring-up vs. branded business/pedagogical configuration) but explicitly not implemented in this pass. No frontend code was written; this document is what future workspace UI implementation is built against.
- **Phase 2 post-implementation stress test** (`docs/ADMINISTRATION_PLATFORM_PLAYBOOK.md`, Phase 2 section — no code changed): the Provider SDK was checked against 19 hypothetical provider families (Entra ID, Apple Sign In, Twilio, WhatsApp, Telegram, Google Maps, Mapbox, Stripe, PayPal, OpenAI, Azure OpenAI, Anthropic, Ollama, RabbitMQ, Kafka, MQTT, S3, Azure Blob, MinIO). 15 of 19 generalize unchanged. Three real, narrow evolution points were recorded as future implementation considerations, none scheduled and none requiring an SDK/Registry/Vault change now: credential sensitivity classification (public/secret/internal), active-provider resolution by capability contract, and provider lifecycle (stateless vs. connection-oriented). Phase 2 remains approved exactly as implemented.
- **Administration Platform architecture amendment pack** (`docs/adr/0018`, `0019`, `0020` — append-only, Decisions 1–7 of ADR-0018 and all prior decisions of ADR-0019/0020 unchanged): closes four findings from a dedicated pre-implementation critical review that deliberately tried to break the frozen architecture rather than validate it. ADR-0018 gains Decision 8 (an optimistic-locking write contract — no write contract existed previously, only `resolve()`'s read path), Decision 9 (`required-permission-to-view`/`-to-edit` are mandatory with no default, closing a default-open privilege gap), and Decision 10 (a registration-time integrity heuristic — dependency fan-out, validation-rule complexity, and Data-Classification/`approval-required` combinations are flagged for human review, finally giving the review's own most-repeated warning, "Business Rules will leak into Configuration because it's faster," a technical trap instead of a paragraph of prose). ADR-0019 gains Decision 5 (Provider Credentials use Blueprint §6's existing cheap Audit tier, never a full diff — closing a real risk that this codebase's own dominant `LogsActivity` convention would otherwise write plaintext secrets into a broadly-readable audit table). A fifth candidate amendment — an automatic, un-overridable approval floor keyed off Data Classification — was proposed, impact-analyzed, and explicitly rejected as more machinery than the underlying problem required; it was merged into ADR-0018 Decision 10 as one trigger condition of the heuristic instead of a separate mechanism. The Administration Platform architecture is now considered final; the next activity is Phase 0 of `docs/ADMINISTRATION_PLATFORM_PLAYBOOK.md`.
- **New standing Sprint completion policy** (`docs/IMPLEMENTATION_PLAYBOOK.md`, "Sprint completion policy" + Definition of Done items 12–14): a sprint is not done, and its Git tag must not be created/moved onto a commit, until tests pass, an explicit ADR compliance review passes, no unresolved architecture finding remains, documentation is updated, and the tag points to that exact approved commit. `v0.2-core-engines` moved from `8b6b357` to `c1f768d` accordingly — the tag now points to the ADR-compliance-fixed commit, not the original Sprint 1.2 commit.

### Fixed
- **ADR compliance review of Sprint 1.3** caught three findings, all fixed before freezing:
  1. **Media's primary key was a plain auto-increment integer**, violating Addendum D4's explicit, named exception for Media ("its primary key should simply be [a ULID] rather than adding a third identifier"). Fixed by making `id` a ULID directly (`database/migrations/2026_07_01_064044_create_media_table.php` now defines `id` via `ulid()->primary()`; the model uses Laravel's `HasUlids` trait) rather than leaving the default int PK plus Spatie's separate `uuid` column as two identifiers doing what Addendum D4 says one should. Spatie's own `uuid` column (used internally by Media Library Pro's JS uploader) was left untouched — it's a distinct, unrelated mechanism.
  2. **The new private-files route was unversioned**, violating Addendum B7 ("`/api/v1` from day one"). Moved under `/api/v1/private-files/{media}` — new code added in this sprint has no excuse to compound the gap, even though the pre-existing `/user` route (Sprint 0.1, frozen) still predates that decision and is out of scope for this review.
  3. **The custom path generator omitted the `{tier}` segment** that both `docs/DOMAIN_BLUEPRINT.md` §12 and the Implementation Playbook's Sprint 1.2.1 spec call for literally. Fixed by reinstating the tier (the media's own disk name) as the first path segment, rather than quietly reinterpreting a frozen spec because the tier also happens to be realized by disk/bucket selection.
  
  See `docs/developer/media-architecture.md`.
- **ADR compliance review of Sprint 1.2** caught a real Core-boundary violation: `ApprovalRequest`/`ApprovalStep` had hard `->constrained('users')` foreign keys, a schema-level dependency from Core into what becomes Identity's Foundation table in Phase 2, violating "Core depends on nothing else in the entire system — not even Foundation modules." Fixed by editing the (still-unreleased, unconsumed) Sprint 1.2 migrations directly rather than layering an `ALTER TABLE` migration on top of a known-wrong original — the actor columns now store a User ID by convention, with referential integrity left to the calling module. Same review added `SoftDeletes` to both models, since approval decisions are evidentiary and nothing should allow that trail to silently disappear via a direct `delete()` call, matching the "the record of a decision must survive" principle already established for Identity Maintenance. See `docs/developer/approval-engine.md`.

- **The word "Settings" is retired from the architecture vocabulary.** Administration is organized around five durable questions realized through ten named capabilities (Organizational Identity & Structure, Access Governance, Policy & Configuration Governance, Digital Experience Delivery, Communication & Engagement, Connectivity & Interoperability, Asset & Facility Stewardship, Observability & Health Policy, Governance/Risk/Compliance, Platform Extensibility & Product Lifecycle), read through four axes (Altitude, Resolution Timing, Domain Nature, Administrator Persona/Tenancy Line) — never a flat feature list.

Governed by one constraint: this layer is read-mostly and derived, never a shadow copy of live state.

- `App\Core\Services\NumberGeneratorService` + `number_sequences` table + `App\Core\Models\NumberSequence`: row-locked (`lockForUpdate`) atomic increment, scoped sequences (with non-nullable `''`/`0` sentinels to avoid MySQL's NULL-is-distinct unique-index pitfall), yearly/monthly period reset, `{number}` format-pattern templating. Concurrency safety is *proven*, not just implemented — a genuine dual-connection lock-contention test against real MariaDB (`tests/Feature/Core/NumberGeneratorConcurrencyTest.php`), not a sequential-loop stand-in.
- `App\Core\Services\ApprovalEngine` + `ApprovalRequest`/`ApprovalStep` (polymorphic — the one place in Core polymorphism is correct, since Approval is deliberately shallow, unlike Assignment): sequential all-or-nothing multi-step routing, role-or-user eligibility via a duck-typed `hasRole()` check (no direct Spatie dependency), no-self-approval by default (opt-out, not opt-in).
- `docs/developer/number-generator.md`, `docs/developer/approval-engine.md`.
