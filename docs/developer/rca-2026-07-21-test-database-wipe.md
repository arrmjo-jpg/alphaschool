# RCA: Development Database Repeatedly Wiped by `php artisan test`

**Status:** Root cause confirmed and fixed. **Date:** 2026-07-21. **Severity:** High (silent, repeated data loss in every developer's local environment; no data was ever at risk beyond the local Docker dev database â€” nothing in this incident touched a shared or production system).

## Summary

Every `php artisan test` / `pest` invocation inside the `app` container was silently connecting to the real MariaDB `alphaschool` database â€” the same database the running dev app and browser session use â€” instead of the isolated in-memory SQLite database `phpunit.xml` was written to select. Because `tests/Pest.php` binds Laravel's `RefreshDatabase` trait to every Feature test, and `RefreshDatabase` runs `migrate:fresh` (drop all tables, recreate empty schema) once per PHP process, **every test run dropped and recreated every table in the real dev database**, discarding all seeded/manually-entered data. Any work done between two test runs without an intervening `db:seed` was silently lost. This had been happening on every full-suite run since Docker was introduced (2026-07-20) and was misdiagnosed at the time as an unrelated database-state anomaly.

A second, related finding fell out of fixing this: the "12 pre-existing, separately-tracked test failures" (`activity_log.subject_id` truncation under MariaDB strict mode) referenced throughout this project's prior documentation were not a real product bug. They only existed because the test suite was accidentally running against real MariaDB. With the fix in place, all 367 backend tests pass â€” zero failures. See [Secondary finding](#secondary-finding-the-12-pre-existing-failures-were-never-real) below.

## Root cause (confirmed by live reproduction, not inferred)

Three facts combine to cause the wipe:

1. **`docker-compose.yml`** (`app`, `queue`, `scheduler` services) loads `.env.docker` via `env_file:`, which sets `DB_CONNECTION=mariadb`, `DB_DATABASE=alphaschool`, `CACHE_STORE=redis`, `SESSION_DRIVER=redis`, `QUEUE_CONNECTION=redis`, `MAIL_MAILER=smtp` as **real, process-level environment variables** inside the `app` container â€” present before PHP even starts, in both `$_ENV` and `$_SERVER` (PHP's CLI SAPI mirrors real OS environment variables into both superglobals at process start).

2. **`backend/phpunit.xml`** declared `<env name="DB_CONNECTION" value="sqlite"/>` and `<env name="DB_DATABASE" value=":memory:"/>` (plus similar overrides for cache/session/queue/mail) **without `force="true"`**. PHPUnit's own documented `<env>` semantics (`vendor/phpunit/phpunit/src/TextUI/Configuration/PhpHandler.php::handleEnvVariables()`) is: without `force`, only set the variable if `getenv($name)` is currently `false`. Since Docker's `env_file` had already set these, PHPUnit's overrides were silently no-ops â€” confirmed by a MariaDB-specific `SQLSTATE[01000]` warning (a code SQLite cannot emit) recurring on nearly every test run throughout the session.

3. **`backend/tests/Pest.php`** binds `RefreshDatabase` to the entire `Feature` test suite (`pest()->extend(TestCase::class)->use(RefreshDatabase::class)->in('Feature')`). `RefreshDatabase::refreshTestDatabase()` (`vendor/laravel/framework/.../Testing/RefreshDatabase.php`) unconditionally runs `$this->artisan('migrate:fresh', ...)` the first time it executes within a given PHP process (tracked by the static `RefreshDatabaseState::$migrated` flag, which resets every new `php artisan test` invocation since each is a fresh PHP process). `migrate:fresh` drops every table and recreates the schema empty, then Feature tests wrap subsequent tests in rolled-back transactions â€” but that first `migrate:fresh` call, run against whatever `config('database.default')` resolves to, is what actually erases data. Because of point 2, `config('database.default')` resolved to `mariadb`/`alphaschool`, the real dev database â€” not `sqlite`/`:memory:`.

**Adding `force="true"` alone was not sufficient** â€” a live reproduction after that change still wiped the database. A dedicated diagnostic test proved why: PHPUnit's `force="true"` only overrides `putenv()`/`$_ENV`, never `$_SERVER`, and `Illuminate\Support\Env::getRepository()`'s dotenv repository resolves through an adapter chain that reads `$_SERVER`, which force=true never touched â€” so `$_SERVER['DB_CONNECTION']` still read `mariadb` even with `$_ENV['DB_CONNECTION'] === 'sqlite'`, and Laravel's `config('database.default')` still resolved to `mariadb`. The complete fix required **also** adding matching `<server>` entries (PHPUnit XML's `<server>` tag always overwrites `$_SERVER` unconditionally, no force flag needed or available) for every variable at risk. Both blocks are now present in `phpunit.xml`.

## Live reproduction and verification (what was actually run, not assumed)

All of the following was executed against the running Docker dev environment, in this order, with row counts checked via a direct `mariadb` CLI query before and after each step â€” not inferred from test output:

1. `docker compose exec app php artisan db:seed` â†’ `users = 1`.
2. `docker compose exec app php artisan test tests/Feature/Identity/WorkspaceControllerTest.php` (5 tests, all passed) â†’ `users = 0`. **Isolated proof of causation**: nothing else touched Docker or the database between steps 1 and 2.
3. Added `force="true"` to every `<env>` entry in `phpunit.xml`. Repeated steps 1â€“2 â†’ **still `users = 0` after the test run**. `force="true"` alone did not fix it.
4. Added a temporary diagnostic Pest test dumping `getenv()`, `$_ENV`, `$_SERVER`, and `config('database.default')` for `DB_CONNECTION`. Result: `getenv` and `$_ENV` correctly read `sqlite`; **`$_SERVER` and `config('database.default')` still read `mariadb`**. This pinpointed the exact superglobal split.
5. Added matching `<server>` entries to `phpunit.xml` for the same variable set. Re-ran the diagnostic test â€” all four now agreed: `sqlite`/`:memory:`.
6. Repeated the full reproduction end-to-end via the normal invocation (`docker compose exec app php artisan test tests/Feature/Identity/WorkspaceControllerTest.php`): seeded (`users = 1`) â†’ ran the test â†’ `users = 1`, unchanged. **Confirmed fixed.**
7. Ran the full backend suite: **367 passed, 0 failed**, in 72s (down from the 194â€“354s it previously took while silently round-tripping to a real networked MariaDB instance for every query). Diagnostic test file deleted; dev database reseeded to its normal working state (`db:seed`, `storage:link`, `administration:sync-settings`, `administration:sync-providers`) afterward.

## Checklist against every hypothesis the investigation was asked to rule in or out

- **Exact commands executed during the session that could relate to this**: reconstructed from the full session transcript â€” see the "Timeline" section below. In summary: `docker compose up -d --build` (2026-07-20 07:32), `migrate --force` (once, correctly on an empty new volume), a cluster of `docker compose exec app php artisan test` runs on 07-20 and 07-21 (each one silently wiping the database), interspersed with `db:seed`/`administration:sync-*` runs that repopulated it â€” the empty state discovered mid-session was simply whichever test run happened to be the most recent one before the next reseed.
- **Can any PHPUnit test execute `RefreshDatabase`/`DatabaseMigrations`/`DatabaseTruncation` against the dev database?** Yes â€” confirmed as the proximate mechanism. `RefreshDatabase` (bound to all Feature tests via `Pest.php`) was executing `migrate:fresh` against the real `mariadb` connection, not an isolated one, because of the environment-variable precedence bug above. No test explicitly opts into `DatabaseMigrations`/`DatabaseTruncation`; `RefreshDatabase` alone was sufficient to cause this.
- **Could `.env.testing` or environment detection have pointed tests at the wrong database?** No `.env.testing` file exists anywhere in the repo (checked both on the host and inside the container) â€” Laravel's `LoadEnvironmentVariables::checkForSpecificEnvironmentFile()` looks for one when `APP_ENV=testing` and falls back to plain `.env` when absent, which is what happened here. The real mechanism was not `.env.testing`-related; it was the `env_file`/PHPUnit-XML precedence issue described above, which is a different failure mode than the one this question was checking for.
- **Does any Seeder, migration, sync command, or artisan command truncate tables?** No â€” `grep`'d every seeder (`backend/database/seeders/`) and every console command (`backend/app/Console/Commands/`) for `migrate:fresh`, `migrate:refresh`, `db:wipe`, `truncate(`, `TRUNCATE`, `Schema::drop`: zero matches. Ruled out.
- **Could a Docker lifecycle event have recreated the database volume?** No â€” `mysql_data` is declared as a proper named top-level volume in `docker-compose.yml` (not a bind mount, not anonymous), and the only `docker compose down -v` calls anywhere in the session's transcript were against two unrelated, separately-namespaced throwaway projects (`clonetest`, `clonetest2`) used earlier to verify the Docker bootstrap docs on a clean clone â€” confirmed via their own project-scoped volume/network names, distinct from the main `alphaschool` project. No `down -v` was ever run against the main project. The `migrations` table's own row count (43, single batch) is additional physical evidence *for* `migrate:fresh` and *against* volume recreation â€” a recreated volume would show the same 43-migration signature but so would many other explanations, whereas the batch-1-only pattern combined with the live reproduction is what makes `migrate:fresh` the confirmed mechanism rather than a merely-consistent one.
- **Could manual SQL run during debugging have cleared data?** No â€” every raw SQL statement run via the `mariadb` CLI during the session's original investigation was a read-only `SELECT COUNT(*)` / `SHOW TABLES` (verified against the transcript). No `DELETE`, `TRUNCATE`, or `DROP` was ever issued manually.
- **Remaining hypotheses**: none. The mechanism above was reproduced on demand, three times, with the database wiping exactly when and only when a `RefreshDatabase`-bound test ran against the real MariaDB connection, and not wiping once both the `<env force="true">` and `<server>` blocks were in place. This is treated as fully explained, not merely narrowed down.

## The fix

`backend/phpunit.xml` â€” every variable in the `<php>` block now has **both** an `<env force="true">` entry (so it wins if the container's real environment already set the same key) **and** a matching `<server>` entry (so `$_SERVER` â€” which Laravel's config resolution also reads â€” agrees with it). See the inline comment in `phpunit.xml` itself for the mechanism; this document is the incident record, that comment is the load-bearing warning against silently "cleaning up" one block as apparently redundant with the other in the future.

No application code, seeder, migration, or Docker infrastructure needed to change â€” this was purely a test-configuration precedence bug, invisible until reproduced live because both individual pieces (Docker's `env_file`, `phpunit.xml`'s `<env>` block, `RefreshDatabase`'s `migrate:fresh` call) are each independently correct and well-established Laravel/Docker patterns; the bug only exists in how they compose together under Docker specifically, which is why it never manifested on the pre-Docker Laragon setup.

## Automated guard added (`backend/tests/TestCase.php`)

A documentation-only fix (`phpunit.xml`) is not a hard guarantee against this recurring â€” a future edit to `phpunit.xml` could silently drop the `<server>` block again (it looks redundant with the `<env force="true">` block sitting right above it, which is exactly the trap). `tests/TestCase.php` now overrides `createApplication()` to check `config('database.default')`/`config('database.connections.*.database')` immediately after the application boots and **before** `RefreshDatabase`'s `migrate:fresh` hook can run, throwing a `RuntimeException` naming the actual resolved connection if it isn't `sqlite`/`:memory:`.

This was not a trivial addition â€” the first version hooked into this project's own `setUp()` override (`parent::setUp()` then check), which was proven **too late** by live reproduction: with the `phpunit.xml` fix deliberately disabled again, that version of the guard fired its error correctly, but the dev database had already been wiped by the time it did, because `RefreshDatabase`'s hook runs inside `parent::setUp()`, before any code the subclass's own `setUp()` runs after calling it. Moving the check into `createApplication()` (which Laravel's own `TestCase::refreshApplication()` calls, and which returns strictly before `setUpTraits()` triggers `RefreshDatabase`) was verified, the same way, to stop the wipe entirely: reseeded, disabled the fix again, ran the same test â€” `users` count unchanged, only the guard's `RuntimeException` fired. Re-enabled the fix, reran the full suite: 366/366 passing (one fewer than the 367 above only because the temporary diagnostic test file used during the investigation was already deleted by then).

## Secondary finding: the "12 pre-existing failures" were never real

Every prior phase of this project (Phase E-B, F-A, F-B, and the original Docker milestone itself) documented "12 pre-existing, separately-tracked test failures" caused by `activity_log.subject_id` truncation under MariaDB's strict-mode column typing (a `bigint` morph column receiving a ULID string from `Media`'s primary key). That bug write-up assumed a real, if narrow, schema incompatibility worth fixing separately one day.

With this fix in place, the full suite now passes **367/367, zero failures**. Those 12 failures only existed because the test suite was accidentally running against real MariaDB (which enforces strict-mode column truncation) instead of the intended in-memory SQLite (which does not enforce the same typing strictness) â€” i.e., they were a symptom of this exact root cause, not an independent, real product bug.

**Correction pass complete (2026-07-21):** every prior reference describing this as a real, independent Media/`activity_log` bug has been annotated with a correction note pointing back to this RCA â€” `CHANGELOG.md`'s Docker-milestone entry and Phase F-B entry, `docs/IMPLEMENTATION_PLAYBOOK.md`'s Docker-milestone paragraph and Phase F-B entry, and `docs/ADMIN_DESIGN_SYSTEM.md`'s Phase F-B entry. Per explicit instruction, none of the original text was deleted or rewritten â€” each correction is appended as its own clearly-dated note, so the historical record still shows what was believed and reported at the time, with the current, corrected understanding layered on top rather than replacing it.

## Standing rule added

Per explicit instruction: **before starting any major implementation phase, verify the test environment is isolated from the development database.** The automated guard above (`tests/TestCase.php`) makes this self-enforcing for every test run going forward, but as an explicit pre-flight check before relying on test output during a new or resumed session: run one test and confirm a `SELECT COUNT(*)` against the real dev database is unaffected, or simply trust a clean `php artisan test` run â€” if the guard were ever defeated again, it fails loudly with a `RuntimeException` naming the actual resolved connection, not a silent wipe. This is now codified in `docs/IMPLEMENTATION_PLAYBOOK.md`'s Global Engineering Discipline and in the repo's root `CLAUDE.md`.
