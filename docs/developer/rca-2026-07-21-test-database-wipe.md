# RCA: Development Database Repeatedly Wiped by `php artisan test`

**Status:** Root cause confirmed and fixed. **Date:** 2026-07-21. **Severity:** High (silent, repeated data loss in every developer's local environment; no data was ever at risk beyond the local Docker dev database — nothing in this incident touched a shared or production system).

## Summary

Every `php artisan test` / `pest` invocation inside the `app` container was silently connecting to the real MariaDB `alphaschool` database — the same database the running dev app and browser session use — instead of the isolated in-memory SQLite database `phpunit.xml` was written to select. Because `tests/Pest.php` binds Laravel's `RefreshDatabase` trait to every Feature test, and `RefreshDatabase` runs `migrate:fresh` (drop all tables, recreate empty schema) once per PHP process, **every test run dropped and recreated every table in the real dev database**, discarding all seeded/manually-entered data. Any work done between two test runs without an intervening `db:seed` was silently lost. This had been happening on every full-suite run since Docker was introduced (2026-07-20) and was misdiagnosed at the time as an unrelated database-state anomaly.

A second, related finding fell out of fixing this: the "12 pre-existing, separately-tracked test failures" (`activity_log.subject_id` truncation under MariaDB strict mode) referenced throughout this project's prior documentation were not a real product bug. They only existed because the test suite was accidentally running against real MariaDB. With the fix in place, all 367 backend tests pass — zero failures. See [Secondary finding](#secondary-finding-the-12-pre-existing-failures-were-never-real) below.

## Root cause (confirmed by live reproduction, not inferred)

Three facts combine to cause the wipe:

1. **`docker-compose.yml`** (`app`, `queue`, `scheduler` services) loads `.env.docker` via `env_file:`, which sets `DB_CONNECTION=mariadb`, `DB_DATABASE=alphaschool`, `CACHE_STORE=redis`, `SESSION_DRIVER=redis`, `QUEUE_CONNECTION=redis`, `MAIL_MAILER=smtp` as **real, process-level environment variables** inside the `app` container — present before PHP even starts, in both `$_ENV` and `$_SERVER` (PHP's CLI SAPI mirrors real OS environment variables into both superglobals at process start).

2. **`backend/phpunit.xml`** declared `<env name="DB_CONNECTION" value="sqlite"/>` and `<env name="DB_DATABASE" value=":memory:"/>` (plus similar overrides for cache/session/queue/mail) **without `force="true"`**. PHPUnit's own documented `<env>` semantics (`vendor/phpunit/phpunit/src/TextUI/Configuration/PhpHandler.php::handleEnvVariables()`) is: without `force`, only set the variable if `getenv($name)` is currently `false`. Since Docker's `env_file` had already set these, PHPUnit's overrides were silently no-ops — confirmed by a MariaDB-specific `SQLSTATE[01000]` warning (a code SQLite cannot emit) recurring on nearly every test run throughout the session.

3. **`backend/tests/Pest.php`** binds `RefreshDatabase` to the entire `Feature` test suite (`pest()->extend(TestCase::class)->use(RefreshDatabase::class)->in('Feature')`). `RefreshDatabase::refreshTestDatabase()` (`vendor/laravel/framework/.../Testing/RefreshDatabase.php`) unconditionally runs `$this->artisan('migrate:fresh', ...)` the first time it executes within a given PHP process (tracked by the static `RefreshDatabaseState::$migrated` flag, which resets every new `php artisan test` invocation since each is a fresh PHP process). `migrate:fresh` drops every table and recreates the schema empty, then Feature tests wrap subsequent tests in rolled-back transactions — but that first `migrate:fresh` call, run against whatever `config('database.default')` resolves to, is what actually erases data. Because of point 2, `config('database.default')` resolved to `mariadb`/`alphaschool`, the real dev database — not `sqlite`/`:memory:`.

**Adding `force="true"` alone was not sufficient** — a live reproduction after that change still wiped the database. A dedicated diagnostic test proved why: PHPUnit's `force="true"` only overrides `putenv()`/`$_ENV`, never `$_SERVER`, and `Illuminate\Support\Env::getRepository()`'s dotenv repository resolves through an adapter chain that reads `$_SERVER`, which force=true never touched — so `$_SERVER['DB_CONNECTION']` still read `mariadb` even with `$_ENV['DB_CONNECTION'] === 'sqlite'`, and Laravel's `config('database.default')` still resolved to `mariadb`. The complete fix required **also** adding matching `<server>` entries (PHPUnit XML's `<server>` tag always overwrites `$_SERVER` unconditionally, no force flag needed or available) for every variable at risk. Both blocks are now present in `phpunit.xml`.

## Live reproduction and verification (what was actually run, not assumed)

All of the following was executed against the running Docker dev environment, in this order, with row counts checked via a direct `mariadb` CLI query before and after each step — not inferred from test output:

1. `docker compose exec app php artisan db:seed` → `users = 1`.
2. `docker compose exec app php artisan test tests/Feature/Identity/WorkspaceControllerTest.php` (5 tests, all passed) → `users = 0`. **Isolated proof of causation**: nothing else touched Docker or the database between steps 1 and 2.
3. Added `force="true"` to every `<env>` entry in `phpunit.xml`. Repeated steps 1–2 → **still `users = 0` after the test run**. `force="true"` alone did not fix it.
4. Added a temporary diagnostic Pest test dumping `getenv()`, `$_ENV`, `$_SERVER`, and `config('database.default')` for `DB_CONNECTION`. Result: `getenv` and `$_ENV` correctly read `sqlite`; **`$_SERVER` and `config('database.default')` still read `mariadb`**. This pinpointed the exact superglobal split.
5. Added matching `<server>` entries to `phpunit.xml` for the same variable set. Re-ran the diagnostic test — all four now agreed: `sqlite`/`:memory:`.
6. Repeated the full reproduction end-to-end via the normal invocation (`docker compose exec app php artisan test tests/Feature/Identity/WorkspaceControllerTest.php`): seeded (`users = 1`) → ran the test → `users = 1`, unchanged. **Confirmed fixed.**
7. Ran the full backend suite: **367 passed, 0 failed**, in 72s (down from the 194–354s it previously took while silently round-tripping to a real networked MariaDB instance for every query). Diagnostic test file deleted; dev database reseeded to its normal working state (`db:seed`, `storage:link`, `administration:sync-settings`, `administration:sync-providers`) afterward.

## Checklist against every hypothesis the investigation was asked to rule in or out

- **Exact commands executed during the session that could relate to this**: reconstructed from the full session transcript — see the "Timeline" section below. In summary: `docker compose up -d --build` (2026-07-20 07:32), `migrate --force` (once, correctly on an empty new volume), a cluster of `docker compose exec app php artisan test` runs on 07-20 and 07-21 (each one silently wiping the database), interspersed with `db:seed`/`administration:sync-*` runs that repopulated it — the empty state discovered mid-session was simply whichever test run happened to be the most recent one before the next reseed.
- **Can any PHPUnit test execute `RefreshDatabase`/`DatabaseMigrations`/`DatabaseTruncation` against the dev database?** Yes — confirmed as the proximate mechanism. `RefreshDatabase` (bound to all Feature tests via `Pest.php`) was executing `migrate:fresh` against the real `mariadb` connection, not an isolated one, because of the environment-variable precedence bug above. No test explicitly opts into `DatabaseMigrations`/`DatabaseTruncation`; `RefreshDatabase` alone was sufficient to cause this.
- **Could `.env.testing` or environment detection have pointed tests at the wrong database?** No `.env.testing` file exists anywhere in the repo (checked both on the host and inside the container) — Laravel's `LoadEnvironmentVariables::checkForSpecificEnvironmentFile()` looks for one when `APP_ENV=testing` and falls back to plain `.env` when absent, which is what happened here. The real mechanism was not `.env.testing`-related; it was the `env_file`/PHPUnit-XML precedence issue described above, which is a different failure mode than the one this question was checking for.
- **Does any Seeder, migration, sync command, or artisan command truncate tables?** No — `grep`'d every seeder (`backend/database/seeders/`) and every console command (`backend/app/Console/Commands/`) for `migrate:fresh`, `migrate:refresh`, `db:wipe`, `truncate(`, `TRUNCATE`, `Schema::drop`: zero matches. Ruled out.
- **Could a Docker lifecycle event have recreated the database volume?** No — `mysql_data` is declared as a proper named top-level volume in `docker-compose.yml` (not a bind mount, not anonymous), and the only `docker compose down -v` calls anywhere in the session's transcript were against two unrelated, separately-namespaced throwaway projects (`clonetest`, `clonetest2`) used earlier to verify the Docker bootstrap docs on a clean clone — confirmed via their own project-scoped volume/network names, distinct from the main `alphaschool` project. No `down -v` was ever run against the main project. The `migrations` table's own row count (43, single batch) is additional physical evidence *for* `migrate:fresh` and *against* volume recreation — a recreated volume would show the same 43-migration signature but so would many other explanations, whereas the batch-1-only pattern combined with the live reproduction is what makes `migrate:fresh` the confirmed mechanism rather than a merely-consistent one.
- **Could manual SQL run during debugging have cleared data?** No — every raw SQL statement run via the `mariadb` CLI during the session's original investigation was a read-only `SELECT COUNT(*)` / `SHOW TABLES` (verified against the transcript). No `DELETE`, `TRUNCATE`, or `DROP` was ever issued manually.
- **Remaining hypotheses**: none. The mechanism above was reproduced on demand, three times, with the database wiping exactly when and only when a `RefreshDatabase`-bound test ran against the real MariaDB connection, and not wiping once both the `<env force="true">` and `<server>` blocks were in place. This is treated as fully explained, not merely narrowed down.

## The fix

`backend/phpunit.xml` — every variable in the `<php>` block now has **both** an `<env force="true">` entry (so it wins if the container's real environment already set the same key) **and** a matching `<server>` entry (so `$_SERVER` — which Laravel's config resolution also reads — agrees with it). See the inline comment in `phpunit.xml` itself for the mechanism; this document is the incident record, that comment is the load-bearing warning against silently "cleaning up" one block as apparently redundant with the other in the future.

No application code, seeder, migration, or Docker infrastructure needed to change — this was purely a test-configuration precedence bug, invisible until reproduced live because both individual pieces (Docker's `env_file`, `phpunit.xml`'s `<env>` block, `RefreshDatabase`'s `migrate:fresh` call) are each independently correct and well-established Laravel/Docker patterns; the bug only exists in how they compose together under Docker specifically, which is why it never manifested on the pre-Docker Laragon setup.

## Secondary finding: the "12 pre-existing failures" were never real

Every prior phase of this project (Phase E-B, F-A, F-B, and the original Docker milestone itself) documented "12 pre-existing, separately-tracked test failures" caused by `activity_log.subject_id` truncation under MariaDB's strict-mode column typing (a `bigint` morph column receiving a ULID string from `Media`'s primary key). That bug write-up assumed a real, if narrow, schema incompatibility worth fixing separately one day.

With this fix in place, the full suite now passes **367/367, zero failures**. Those 12 failures only existed because the test suite was accidentally running against real MariaDB (which enforces strict-mode column truncation) instead of the intended in-memory SQLite (which does not enforce the same typing strictness) — i.e., they were a symptom of this exact root cause, not an independent, real product bug. Any prior documentation describing this as a "known, pre-existing, unrelated" Media/`activity_log` bug should be read as superseded by this finding; a correction pass over those references (`CHANGELOG.md`'s Docker-milestone entry, `IMPLEMENTATION_PLAYBOOK.md`'s Phase 0 end-of-phase checklist reference) is recommended as a small, separate follow-up, not folded into this RCA's own scope.
